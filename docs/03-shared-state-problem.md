# 03. 공유 상태 문제와 해결 방식

이 문서는 이 저장소가 다루는 핵심 문제를 한 번 더 정리합니다.

## 기존 문제: 화면마다 같은 모델이 분산되던 방식

구형 예제의 핵심은 “두 화면이 같은 `Article`을 공유”합니다.
대표 동작은 아래와 같습니다.

- 목록 화면에서 항목을 본다
- 항목을 탭해 상세로 이동
- 상세에서 수정
- 저장 후 목록이 즉시 반영되어야 함

문제는 이 동작을 상태를 여러 곳에서 흩어져 다루면 쉽게 깨진다는 점입니다.

- 목록과 상세가 각자 `Article` 인스턴스를 들고 있으면 동기화 누락
- “어느 화면이 최신값인지”를 판단하기 어려움
- 삭제/선택 같은 경계 동작에서 상태 불일치 가능

## 해결 전략: 단일 진실 원천(Single Source of Truth)

이 프로젝트에서는 상태를 하나로 모읍니다.

- 목록과 상세는 각각 `Article`을 독립적으로 소유하지 않음
- 현재 선택 항목은 `selectedArticleId`로 통제
- 목록 데이터는 `articles` 한 군데만 갱신

구조를 단순화하면 다음이 보장됩니다.

- 상세에서 수정을 눌러도 목록 데이터 갱신 경로가 한 곳
- 저장/삭제 후에도 UI가 같은 상태를 공유해 재렌더링됨
- 경계 동작(삭제 후 선택 초기화)도 동일 규칙으로 처리 가능

## 동작 예시(현재 코드 기준)

1. 사용자: 목록 항목 클릭
2. `ViewModel.selectArticle(id)` 호출
3. `ClearSelection`/`UpdateArticle`/`DeleteArticle` 같은 액션으로 상태 변경
4. `ArticleListReducer`에서 새 상태 계산
5. `uiState` 변경
6. Compose가 재조합되어 목록 또는 상세 화면 렌더링

## 경계 케이스

### 선택된 글 삭제

삭제 액션에서 삭제 대상이 현재 선택 항목이면 `selectedArticleId`를 `null`로 바꿔 상세 화면에서 다시 목록으로 복귀되도록 처리합니다.

### 상세 진입/퇴장 시점 정리

상세 화면으로 이동/종료할 때 `ClearSelection`을 통해 `selectedArticleId`를 초기화합니다.
이 전략은 상태 경로를 단순화해 “모르는 상태의 잔류 선택값” 버그를 줄입니다.

## 왜 중요한가

이 문제를 먼저 문서화하면 나중에 `Article` 도메인을 확장해도 공통 규칙을 재사용할 수 있습니다.  
즉, 화면 수가 늘어나도 상태 충돌이 생기는 위치가 줄어듭니다.
